# How It Works — AI Guidelines System

This document explains how the AI Guidelines system works **for the AI agent**. Read this first to understand the relationship between all components.

---

## System Overview

```text
project-root/
├── CLAUDE.md                  ← Main config file. Source of truth for standards, philosophy, and project-specific settings.
├── HOW_IT_WORKS.md            ← This file. System meta-documentation.
├── .agents/                   ← Agent intelligence layer
│   ├── skills/                ← Specialized executors (SKILL.md files)
│   ├── workflows/             ← Orchestrators (slash command .md files)
│   ├── rules/                 ← Always-on enforcement rules
│   └── scripts/               ← Helper shell scripts
├── .claude/                   ← Claude Code-specific mirror (synced with .agents/)
│   └── skills/                ← Same skills as .agents/skills/
├── docs/                      ← Project documentation (generated and manual)
│   ├── engineering/           ← Code standards, testing, workflow, quality gates
│   ├── architecture/          ← System architecture (generated by /generate-architecture)
│   ├── design/                ← UI/UX specs (generated by /generate-ui-design)
│   ├── tasks/                 ← Task breakdowns (generated by /generate-task-breakdown)
│   ├── progress/              ← Progress tracking (maintained by implement-task)
│   └── AGENT_FLOW.md          ← Canonical lifecycle doc: routing, pipeline, gates
└── .ai-guidelines.json        ← Manifest file (installed version, file hashes, locale)
```

## Skills vs Workflows vs Rules

| Component    | Purpose                                                 | Activation                                       | Example                                        |
| ------------ | ------------------------------------------------------- | ------------------------------------------------ | ---------------------------------------------- |
| **Skill**    | Specialized executor with detailed steps                | Matched by `description` in SKILL.md frontmatter | `implement-task`, `bug-fixer`, `generate-test` |
| **Workflow** | Orchestrator that sequences steps and may invoke skills | Slash command (`/helpme`, `/code-review`)        | `/debug` invokes `bug-fixer` skill             |
| **Rule**     | Always-on enforcement applied to every interaction      | `trigger: always_on` in frontmatter              | `00-bootstrap.md` (context loading)            |

### Key Relationships

- **Workflows call skills**, not the other way around
- **Rules apply to everything** — they wrap around both skills and workflows
- **Skills are self-contained** — each has its own `SKILL.md` with complete instructions
- **`.agents/` and `.claude/` must stay in sync** — both contain the same skills

## Discovery Mechanism

When the agent receives a task:

1. **Rules fire first** (`00-bootstrap.md`) — loads `CLAUDE.md`, checks for relevant skills
2. **Intent matching** — the agent matches user intent to closest skill/workflow via `description` field in frontmatter
3. **If workflow matches** — follow the workflow steps (which may invoke skills)
4. **If skill matches** — follow the SKILL.md instructions directly
5. **If nothing matches** — fallback to `CLAUDE.md` + existing code patterns

The routing table in `docs/AGENT_FLOW.md` §2 maps common intents to skills/workflows.

## Quality Gates Flow

Quality gates are defined in `docs/engineering/QUALITY_GATES.md` (canonical source).

```text
Planning ──▶ Security Threat Analysis + i18n Planning (§3.1 of AGENT_FLOW.md)
     │
     ▼
Code Change ──▶ Lint ──▶ Test (3 pillars) ──▶ i18n Check ──▶ Security Scan ──▶ Commit
     │
     ▼
Pre-PR ──▶ /code-review workflow ──▶ PR
```

Gates always execute. No exceptions without user confirmation.

## Lifecycle of a Task

Complete trace from user request to commit:

1. **User says**: "Implement feature X"
2. **Rule `00-bootstrap.md`** fires → agent reads `CLAUDE.md`, searches for matching skill
3. **Match found**: `task-planner` skill (plans) → routes to `implement-task` (executes)
4. **`implement-task` runs**:
   - Reads task spec (from `docs/tasks/`, issue, or user input)
   - Creates semantic branch (`feat/feature-x`)
   - Implements code following `CODE_STANDARDS.md`
   - Runs quality gates (Lint → Test → i18n → Security)
   - Updates progress tracking (`docs/progress/`)
   - Makes semantic commits
5. **Rule `02-transparency.md`** enforces → agent reports decisions and sources
6. **User asks for PR** → `/code-review` workflow → PR opened

## Adaptation via init-project

The `init-project` skill adapts guidelines to the target project:

1. **Detects** project type, stack, versions, packages, database
2. **Asks** guided questions about tech decisions (via questionnaire)
3. **Updates** `CLAUDE.md` with project-specific settings
4. **Updates** `docs/engineering/STACK.md` with detected versions
5. **Identifies** missing stack packs and suggests activation

This ensures guidelines match the actual project, not assumptions.

## Extending the System

### Adding a New Skill

Use the `skill-creator` skill or manually:

1. Create directory: `.agents/skills/<skill-name>/`
2. Create `SKILL.md` with frontmatter (`name`, `description`, `dependencies`)
3. Optionally add `references/` for supplementary docs
4. Mirror to `.claude/skills/<skill-name>/`
5. Update `AGENT_FLOW.md` routing table if the skill handles common intents

### Adding a New Workflow

1. Create `.agents/workflows/<workflow-name>.md`
2. Add YAML frontmatter with `description`
3. Define steps that may invoke skills
4. Use `// turbo` annotation for auto-runnable steps

### Adding a Stack Pack

1. Create `.agents/skills/stack-<name>/SKILL.md`
2. Define: standards, linting, testing, quality gates, patterns
3. Mirror to `.claude/skills/`
4. Update `init-project` detection rules
5. Update `verify-quality-gates.sh` if new quality gate commands needed
